# 第12章 线程同步 ⭐⭐⭐⭐(多线程重点,面试问得最深)

> 通用规则:
> - 5种机制都是一个套路: **init → 加锁/获取 → 临界区 → 解锁/释放 → destroy**
> - 区别只在"拿不到锁时的行为"和"什么场景用"
> - pthread系列(mutex/cond/spin/rwlock)返回错误码;但**信号量 sem_* 是 -1+errno**(不同!)
> - 编译加 `-lpthread`

---

## 12.1 同步概念 + 竞态演示 ⭐⭐⭐

**竞态条件(race condition)**: 多线程同时访问共享数据,结果取决于线程的执行时序 → 不确定、不可复现。
典型是"读-改-写"被打断:
```c
l_count = g_count;   // 读
l_count++;           // 改   ← 这三步不是原子的,中途可被另一线程插队
g_count = l_count;   // 写
```

### 🔴 demo 1 实录 (ch12/01_mutex.c)
两个线程各给 g_count +1 共 1000 万次,期望 2000 万:
```
不加锁 → 1353万 / 1229万 / 1456万  (每次不同,丢几百万次更新)
加  锁 → 2000万 / 2000万 / 2000万  (稳定正确)
```
**丢更新原理**: A读到100没写回,B也读到100,两个都写101 → 两次+1只生效一次。
**踩的坑**:
```
① mutex 要定义成【全局】,否则线程函数看不见
② "读-改-写"必须在循环【内】每次都做,否则演示不出竞态
③ 线程函数返回 void*;错误检查查【pthread_create的返回值】不是tid
④ strerror 要 #include <string.h>
```

## 12.2 互斥锁 mutex ⭐⭐⭐⭐(最常用)

### API
```c
pthread_mutex_t mutex;
pthread_mutex_init(&mutex, NULL);    // 动态初始化
// 或静态: pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_lock(&mutex);          // 加锁,拿不到就【睡眠】等
pthread_mutex_unlock(&mutex);        // 解锁
pthread_mutex_destroy(&mutex);       // 销毁
```
- 把"读-改-写"夹在 lock/unlock 之间。**临界区尽量小**(只包真正碰共享数据的代码)。

### trylock(非阻塞版)
```c
int pthread_mutex_trylock(&mutex);   // 拿到→0;拿不到→立刻返回EBUSY(不阻塞)
```

### 互斥锁的 4 种类型(了解)
```
PTHREAD_MUTEX_NORMAL      普通(默认行为):重复加锁会死锁
PTHREAD_MUTEX_ERRORCHECK  检错:重复加锁返回错误而非死锁
PTHREAD_MUTEX_RECURSIVE   递归锁:同一线程可重复加锁(加几次解几次)
PTHREAD_MUTEX_DEFAULT     默认(Linux下等于NORMAL)
```

## 12.3 条件变量 cond ⭐⭐⭐⭐(生产者-消费者)

**解决的问题**: 消费者等"库存>0"才能取货。
```
❌ 死循环 while(g_avail<=0){} 空转查 → 狂烧CPU(忙等)
✅ 条件变量: 空仓就【睡着等】,生产者放货时【叫醒】它 → CPU占用0
```

### API
```c
pthread_cond_t cond;
pthread_cond_init(&cond, NULL);              // 或 PTHREAD_COND_INITIALIZER
pthread_cond_wait(&cond, &mutex);            // 等待(传 cond + mutex 两个!)
pthread_cond_signal(&cond);                  // 唤醒一个等待者
pthread_cond_broadcast(&cond);               // 唤醒全部等待者
pthread_cond_destroy(&cond);
```

### 标准用法(背这个骨架)
```c
// 消费者:
pthread_mutex_lock(&mutex);
while (g_avail <= 0)                 // ⭐用 while 不用 if
    pthread_cond_wait(&cond, &mutex);
g_avail--;                           // 消费
pthread_mutex_unlock(&mutex);

// 生产者:
pthread_mutex_lock(&mutex);
g_avail++;                           // 生产
pthread_mutex_unlock(&mutex);
pthread_cond_signal(&cond);          // 叫醒消费者
```

### ⭐ 两个必懂点(面试高频)
**1. cond_wait 为什么要传 mutex?** 它内部原子地做三件事:
```
① 解开 mutex(让生产者能拿锁去生产+signal)
② 让本线程睡着
③ 被唤醒后重新加上 mutex 才返回
→ 调用前必须已持有锁,醒来后仍持有锁
```
**2. 为什么用 while 不用 if?**
```
被唤醒不保证条件成立(虚假唤醒 / 多消费者被别人抢先)
→ while 重新检查才安全;if 可能在库存还是0时往下走 → 出bug
口诀: cond_wait 永远放在 while 里
```

### 🔴 demo 2 实录 (ch12/02_cond.c)
生产者放5个、消费者取5个:
```
无sleep → 生产者太快,先全生产再全消费(但消费正好5个,逻辑对)
有sleep(1) → 整齐交替: 生产1→消费1→消费者睡1秒...
```
**关键观念**: 多线程执行顺序由调度器决定,**不确定**;sleep只是为了肉眼看清节奏,与正确性无关。
**踩的坑**:
```
① cond_wait 写在了锁外面 → 必须 lock 后再 while+cond_wait
② 结构写成两个while → 应是一个 for 循环消费5次
③ main 漏 join → 生产者跑完就退出,消费者被提前杀掉
④ 漏 sleep(1) → 看不出交替(非bug)
```

## 12.4 自旋锁 spinlock ⭐⭐(眼熟即可)

```c
pthread_spinlock_t spin;
pthread_spin_init(&spin, PTHREAD_PROCESS_PRIVATE);
pthread_spin_lock(&spin);
pthread_spin_unlock(&spin);
pthread_spin_destroy(&spin);
```
**和 mutex 唯一区别——拿不到锁时**:
```
mutex:    睡眠,让出CPU
spinlock: 忙等,原地空转反复试,不让CPU
```
- 适用: 锁持有**极短** + **多核**。省掉睡眠/唤醒开销。
- 单核别用(自旋者占CPU,持锁者反而跑不完)。持锁时不能睡眠/阻塞。

## 12.5 读写锁 rwlock ⭐⭐(眼熟即可)

```c
pthread_rwlock_t rw;
pthread_rwlock_init(&rw, NULL);
pthread_rwlock_rdlock(&rw);   // 读锁:共享(多读者可同时)
pthread_rwlock_wrlock(&rw);   // 写锁:独占
pthread_rwlock_unlock(&rw);   // 读写都用这个解锁
pthread_rwlock_destroy(&rw);
```
```
读+读 → 并发     读+写 → 互斥     写+写 → 互斥
```
- 适用: **读多写少**(配置/缓存),并发度比 mutex 高。
- 坑: 读者太多时写者可能**饿死**。

## 12.6 信号量 semaphore ⭐⭐(眼熟即可)

```c
#include <semaphore.h>          // 注意:不是 pthread.h
sem_t sem;
sem_init(&sem, 0, N);           // 初值N=资源数; 第2参 0=线程间
sem_wait(&sem);                 // P: 计数-1; 为0则阻塞
sem_post(&sem);                 // V: 计数+1; 唤醒等待者
sem_destroy(&sem);
```
- 本质是**计数器** = 可用资源数。N=1→退化成互斥锁;N>1→最多N个线程同时进。
- ⚠️ 在 `<semaphore.h>`;`sem_*` 失败返回 **-1+errno**(和 pthread 风格不同)。

## ⭐ 主对比表(核心,必背)

| 机制 | 解决 | 拿不到时 | 适用场景 |
|------|------|---------|---------|
| **mutex 互斥锁** | 互斥 | 睡眠 | 通用临界区保护 |
| **spinlock 自旋锁** | 互斥 | 忙等 | 锁极短 + 多核 |
| **rwlock 读写锁** | 互斥 | 读共享/写独占 | 读多写少 |
| **cond 条件变量** | 同步 | 睡着等唤醒 | 生产者-消费者 |
| **semaphore 信号量** | 同步+互斥 | 计数为0则等 | 限流/资源池/通知 |

## ⭐ 死锁 deadlock(面试超高频)

两线程**加锁顺序相反**就会死锁:
```c
线程A: lock(m1); lock(m2);   // 先1后2 → 拿着1等2
线程B: lock(m2); lock(m1);   // 先2后1 → 拿着2等1
→ 互相等,卡死
```

**死锁 4 个必要条件**:
```
① 互斥        资源一次只能一个线程占
② 持有并等待  占着A还要等B
③ 不可剥夺    别人不能抢走它的锁
④ 循环等待    A等B、B等A 成环
```
**避免**(破坏任一条件,最常用第④):
```
✅ 固定加锁顺序(所有线程都先1后2) ← 最常用
✅ trylock 拿不到就退避重试
✅ 一次性申请所有锁
```

## 面试速查

- [ ] 什么是竞态条件?(多线程访问共享数据,结果依赖时序,不确定)
- [ ] mutex vs spinlock?(拿不到锁:mutex睡眠让CPU,spinlock忙等空转;自旋适合临界区极短+多核)
- [ ] 读写锁啥时候用?(读多写少,多读者可并发)
- [ ] 信号量 vs 互斥锁?(互斥锁有所有权/值非0即1/谁锁谁解;信号量是计数器/可>1/可被别的线程post)
- [ ] 条件变量为什么配互斥锁?(保护条件判断 + cond_wait 内部要解锁/重锁)
- [ ] cond_wait 为什么放在 while 里?(防虚假唤醒/被别人抢先)
- [ ] 死锁4条件 + 怎么避免?(互斥/持有等待/不可剥夺/循环等待;固定加锁顺序最常用)

## 一句话总结

> **5种同步机制同一个套路(init→加锁→临界区→解锁→destroy),区别在"拿不到锁的行为"和"用在哪":**
> **mutex(睡眠,通用)、spinlock(忙等,极短锁)、rwlock(读共享写独占,读多写少)、cond(睡等唤醒,生产者消费者)、semaphore(计数器,限流/资源池)。**
> **死锁靠"固定加锁顺序"避免。**
