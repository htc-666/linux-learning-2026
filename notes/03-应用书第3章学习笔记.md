## 3.1 Linux 系统如何管理文件

### 3.1.1 静态文件 vs 动态文件 vs inode

#### 三个层次概念

| 概念 | 在哪 | 谁的 |
|---|---|---|
| **静态文件** | 磁盘上 | 文件系统持久存储 |
| **动态文件 / 内核缓冲** | 内存里 | 内核给进程用的缓存 |
| **inode** | 磁盘 inode 区 | 文件元数据 |

#### 磁盘存储层次(自底向上)

```
扇区 Sector (512 字节,硬件最小单位)
  ↓ 8 个组成
块 Block (4KB,文件系统读写最小单位)
  ↓ 多个组成
文件 File 的数据
```

#### inode 的本质

- 磁盘格式化时分两个区:**inode 区** + **数据区**
- 每个文件对应**唯一一个** inode(结构体)
- inode 里**不存文件名**,存:大小、所有者、权限、时间戳、类型、**数据块位置**
- 文件名存在**目录项**里(后面学到第 5 章会讲)
- 查看命令:`ls -i file` 或 `stat file`

#### open 一个文件,系统做了三步

1. 根据文件名查目录,**找到对应 inode 编号**
2. 通过 inode 编号去 inode table 找到**对应 inode 结构体**
3. 根据 inode 中记录的 block 位置,**把数据读出**

#### 我的理解:为什么 Windows 快速格式化很快?

只删 inode 表,**数据区原封不动**。所以用恢复软件能找回数据——它直接扫描数据区找文件特征。这也是为什么处理隐私 U 盘要"低级格式化"或填充随机数据。

### 3.1.2 文件打开时的状态(超重要!)

open 之后,内核会:
1. 申请一段**内存(动态文件缓冲)**
2. 把磁盘上的静态文件**读到这段内存**
3. 你的 read / write **操作的是这份内存副本**,不是直接操作磁盘
4. 后续由内核异步同步回磁盘

#### 为什么这么设计?

| 直接操作磁盘 | 缓冲到内存 |
|---|---|
| 慢(磁盘 IO 比内存慢百万倍) | 快 |
| 必须整块读写 | 可以按字节随机操作 |
| 每次 IO 都阻塞 | 内核可以批量合并写回 |

**这就是为什么"突然断电会丢数据"** —— 你以为已经 write 进文件,实际还在内存缓冲里,没真正写回磁盘。需要调 `sync()` 或 `fsync()` 强制刷盘。

#### 数据结构关系图(三级表,面试必考)

```
┌─────────────────┐
│   进程 PCB       │   ← 内核为每个进程维护
│  (Process       │
│   Control Block)│
└────────┬────────┘
         ↓
┌─────────────────┐
│ 文件描述符表 fd  │   ← 进程私有,数组,索引就是 fd
│ ┌────┐          │
│ │ 0  │  → stdin │
│ │ 1  │  → stdout│
│ │ 2  │  → stderr│
│ │ 3  │  → ?     │
│ └────┘          │
└────────┬────────┘
         ↓
┌─────────────────┐
│   文件表 (file  │   ← 内核全局,记录:
│   table)        │     · 文件状态标志(O_RDONLY 等)
│                 │     · 当前读写偏移量
│                 │     · 引用计数
│                 │     · i-node 指针
└────────┬────────┘
         ↓
┌─────────────────┐
│   inode 节点    │   ← 磁盘上文件元数据
│                 │     一个文件 → 一个 inode
└─────────────────┘
```

**关键认知**:
- **fd 是进程级**(不同进程的 fd 3 可能指向完全不同的文件)
- **文件表是系统级**(多个 fd 可以指向同一个文件表)
- **inode 是磁盘级**(全系统唯一)

这张图是后面 dup、文件共享、并发竞争的理论基础。

## 3.2 返回错误处理与 errno

### errno 是什么?

- 一个**全局变量**(实际是线程局部变量,每个线程独立)
- 类型:`int`
- 头文件:`<errno.h>`
- 当 syscall / libc 函数失败时,内核/libc 会设置 errno
- **下一次错误会覆盖上一次**,所以发现失败要立即检查

### 重要 errno 值

| 宏 | 含义 |
|---|---|
| `ENOENT` | No such file or directory(不存在) |
| `EACCES` | Permission denied(没权限) |
| `EISDIR` | Is a directory(是目录不是文件) |
| `EEXIST` | File exists(O_EXCL 时常见) |
| `ENOMEM` | Out of memory(内存不足) |
| `EINTR` | Interrupted system call(被信号打断,常见!) |
| `EAGAIN` / `EWOULDBLOCK` | 资源暂时不可用(非阻塞 IO 必须处理) |

### 3.2.1 strerror —— 把 errno 翻译成字符串

```c
#include <string.h>
char *strerror(int errnum);
```

用法:

```c
if (open(...) == -1) {
    printf("Error: %s\n", strerror(errno));
    // 输出:Error: No such file or directory
}
```

### 3.2.2 perror —— 一步打印错误

```c
#include <stdio.h>
void perror(const char *s);
```

用法:

```c
if (open(...) == -1) {
    perror("open test_file");
    // 输出:open test_file: No such file or directory
}
```

### 我的选择:用 perror

- 自带 errno 翻译
- 不用手动包含 string.h
- 输出格式标准(自己的描述: errno描述)
- **专业代码几乎全用 perror**

### ⚠ 重要陷阱(我踩过)

```c
if (fd == -1) perror("open");  // ✓ 错误路径
perror("open success");          // ✗ 错误!成功路径用 printf
```

`perror` 不管成功失败,**永远去读当前 errno**。成功路径下 errno 可能是上次的残留值,perror 输出乱码。

## 3.3 退出程序:exit、_exit、_Exit

### 三种退出方式

| 方式 | 类型 | 是否清理 stdio 缓冲 | 是否调注册的 atexit 函数 |
|---|---|---|---|
| `return` (main 中) | 关键字 | ✓ | ✓ |
| `exit()` | C 库函数 | ✓ | ✓ |
| `_exit()` | 系统调用 | ✗ | ✗ |
| `_Exit()` | 系统调用 | ✗ | ✗ |

### 函数签名

```c
#include <stdlib.h>
void exit(int status);     // C 库,会清理

#include <unistd.h>
void _exit(int status);    // syscall,立即退出

#include <stdlib.h>
void _Exit(int status);    // syscall,和 _exit 等价
```

### 关键区别(面试常考)

```c
int main() {
    printf("hello");   // 没有 \n,在 stdio 缓冲区里
    _exit(0);          // ❌ "hello" 不会被打印!缓冲区没刷
    // 改成 exit(0)   ✓ "hello" 会打印,exit 会刷缓冲
}
```

**这就是为什么 `printf` 后用 `_exit` 可能丢输出,用 `exit` 不丢**。

### 推荐用 exit()

99% 场景用 `exit(status)`。**只有在 fork 后子进程异常退出**,才用 `_exit()`(避免重复刷父进程也共享的缓冲区)——这个后面学进程时会用到。

### 状态码约定

- `0` = 正常退出
- 非 0(通常 1 或 -1)= 异常退出
- shell 里 `echo $?` 可以查看上个程序的退出码

## 3.4 空洞文件(Hole File)

### 概念

用 `lseek` 把文件指针跳到**超过当前文件大小**的位置,然后 `write` 数据。中间没写的部分就是**空洞**。

```c
int fd = open("hole_file", O_WRONLY|O_CREAT|O_TRUNC, 0644);
lseek(fd, 4096, SEEK_SET);   // 跳到 4KB 处
write(fd, "X", 1);           // 在 4KB 处写 1 字节
```

文件**逻辑大小**:4097 字节
文件**物理大小**:可能只占用 4KB 块(空洞不占物理空间!)

### 验证

```bash
ls -l hole_file       # 显示 8K(逻辑大小)
du -h hole_file       # 显示 4K(实际占用)
```

**`ls` 和 `du` 显示不一样,就是有空洞**。

### 空洞读出来是什么?

读空洞部分得到**全 0**(`\0`)。

### 实际应用

1. **多线程下载工具**(迅雷):创建一个目标大小的稀疏文件,多个线程同时从不同偏移写入,空洞自动填补
2. **虚拟机磁盘镜像**:VMware/QEMU 的 qcow2、vmdk 都是稀疏文件,实际数据少时不占满分配的空间
3. **预分配文件**:数据库预创建大文件避免后续扩展

### 我的延伸:打类比

修高速公路,5 个工程队同时从不同里程动工,中间路段还没修(空洞),但全长已经规划好。最后所有路段填满,工程结束。

## 3.5 O_APPEND 和 O_TRUNC 标志

### 3.5.1 O_TRUNC

**作用**:open 时如果文件已存在,**清空内容**(长度变 0)。

```c
open("file", O_WRONLY | O_CREAT | O_TRUNC, 0644);
```

**典型场景**:写日志、写配置、写新文件——**几乎永远要加**(否则旧数据残留)。

### 3.5.2 O_APPEND

**作用**:每次 write 前**自动把偏移量移到文件末尾**,实现追加写入。

```c
open("log.txt", O_WRONLY | O_APPEND);
```

**核心特性:O_APPEND 是原子操作**

```
write 的内部行为(有 O_APPEND):
   1. 锁定文件
   2. 把偏移量设到末尾
   3. 写数据
   4. 解锁
   (1-4 是一气呵成,中间不会被打断)
```

这就保证了**多进程同时写同一日志文件,不会互相覆盖**——这正是 syslog / nginx access log 多 worker 写一个文件不乱的原理。

### 重要细节(踩坑预警)

1. **O_APPEND 不影响读**:即使设了 O_APPEND,read 还是从当前偏移开始读
2. **O_APPEND 让 lseek 对写无效**:即使你 `lseek(fd, 0, SEEK_SET)`,下次 write 还是去末尾
3. **O_APPEND + O_TRUNC** 同时用:先清空(O_TRUNC),然后每次写在末尾(此时末尾就是 0)——效果等同 O_TRUNC,O_APPEND 只在文件被多次写入时见效

## 3.6 多次打开同一个文件

### 3.6.1 三个关键现象

#### 现象 1:同一进程内多次 open 得到不同 fd

```c
int fd1 = open("file", O_RDWR);   // fd1 = 3
int fd2 = open("file", O_RDWR);   // fd2 = 4
int fd3 = open("file", O_RDWR);   // fd3 = 5
// 每个都要单独 close
```

**(在 VSCode 终端里跑,fd 可能从 6/7/8 开始,因为 0-5 被 VSCode 占了——所以面试别死记 "新 fd 从 3 开始")**

#### 现象 2:多次 open 同一文件,**内存中只有一份动态文件**

通过 fd1 写入,fd2 立刻能读到(因为它们指的是同一份内存数据)。

```c
write(fd1, "hello", 5);
lseek(fd2, 0, SEEK_SET);
read(fd2, buf, 5);          // buf = "hello"
```

#### 现象 3:不同 fd 的**偏移量相互独立**

```
fd1 → 文件表 1(偏移量 A) ┐
fd2 → 文件表 2(偏移量 B) ├─→ 同一个 inode → 同一份动态文件
fd3 → 文件表 3(偏移量 C) ┘
```

**多次 open = 多个文件表 + 各自偏移 + 同一 inode**

### 3.6.2 多次打开做接续写

两个 fd 想接续写(一个写完另一个接着写),**最简单方法是用 O_APPEND**:

```c
fd1 = open("file", O_RDWR | O_CREAT | O_EXCL | O_APPEND, ...);
fd2 = open("file", O_RDWR | O_APPEND);
// 现在 fd1 和 fd2 写都会自动追加到末尾,顺序写
```

不用 O_APPEND 想接续 → **必定竞争**,见 3.9 节。

### 引用计数

文件表里有个 **引用计数**。每次 close 一个 fd,引用计数 -1。**减到 0,内核才真正释放文件表**。这就是后面 dup 的关键。

## 3.7 复制文件描述符(dup / dup2)

### 与多次 open 的本质区别

| | 多次 open | dup/dup2 |
|---|---|---|
| 文件表 | **多个**,各有偏移 | **共享同一个**,偏移联动 |
| 写入行为 | 互相覆盖(无 O_APPEND) | 接续写(因为共享偏移) |
| inode | 同一个 | 同一个 |

```
dup 后:
fd1 ─┐
     ├─→ 同一个文件表(同一偏移) ─→ inode
fd2 ─┘
```

### 3.7.1 dup —— 系统自动分配新 fd

```c
#include <unistd.h>
int dup(int oldfd);
```

- 成功:返回新 fd(系统分配最小可用值)
- 失败:-1,设置 errno

### 3.7.2 dup2 —— 手动指定新 fd

```c
int dup2(int oldfd, int newfd);
```

- 如果 `newfd` 已被打开,**先 close 它**,再复制
- 这就是 **shell 重定向的核心机制**

### dup2 实现重定向(经典!)

```c
// 实现 ./prog > out.txt 的效果
int fd = open("out.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
dup2(fd, STDOUT_FILENO);     // fd 复制到 1,stdout 现在指向 out.txt
close(fd);

printf("Hello\n");           // 实际写进 out.txt!
```

**面试常问**:"shell 的 `>` `<` `|` 怎么实现?"
**答**:fork 子进程,在 exec 之前用 dup2 把 0/1/2 重定向到文件或管道。

### 一个 fd 可以复制多次

```c
int fd2 = dup(fd1);
int fd3 = dup(fd1);
int fd4 = dup2(fd1, 100);   // 强行用 100
```

### 我的延伸:dup 在驱动开发里的影子

后面学**字符设备驱动**,`copy_to_user` / `copy_from_user` 解决的就是"内核 fd → 用户 fd"数据搬运。fd 在内核里映射到的是 `struct file *`,本质和这里讲的"文件表"是一回事。

## 3.8 文件共享

### 定义

同一个文件被**多个独立读写体**同时操作。

### 三种实现方式

#### 方式 1:同一进程内多次 open

```
PCB
├─ fd1 ─→ 文件表1 ─┐
├─ fd2 ─→ 文件表2 ─┼─→ inode
└─ fd3 ─→ 文件表3 ─┘
```
**多个文件表,偏移独立**。

#### 方式 2:不同进程各自 open

```
进程1.PCB → fd1 → 文件表1 ─┐
                              ├─→ inode (磁盘同一文件)
进程2.PCB → fd2 → 文件表2 ─┘
```
**完全独立的文件表**,适合多进程场景。

#### 方式 3:dup/dup2 / fork 继承

```
进程1.PCB
├─ fd1 ─┐
└─ fd2 ─┴─→ 同一个文件表 ─→ inode
```
**共享文件表,偏移联动**。

### 何时用哪种?

| 场景 | 用哪种 |
|---|---|
| 父子进程写同一日志 | fork 继承(方式 3)+ O_APPEND |
| shell 重定向 | dup2(方式 3) |
| 多线程独立处理文件不同区域 | 多次 open(方式 1) |
| 多个程序读同一配置文件 | 各自 open(方式 2) |

## 3.9 原子操作与竞争冒险 ⭐⭐⭐

### 3.9.1 竞争冒险(Race Condition)

#### 经典场景

进程 A 和 进程 B 都想在文件末尾追加写。**没有 O_APPEND**:

```
时间 →
进程A:  lseek到末尾(1500)   ........... write(fd, buf, 100)
                       ↑                ↑
                  时间片用完         恢复运行
进程B:  ........... lseek到末尾(1500) write(fd, buf, 100) .........
                                                ↑
                                       覆盖了 A 即将要写的位置
```

**结果**:A 的数据被 B 覆盖,或反之。**这就是竞争**。

#### 本质问题

"先定位,再写"是**两步操作**,中间可能被打断。

### 3.9.2 原子操作 —— 三种解法

#### 解法 1:O_APPEND(简单粗暴)

让"定位 + 写"在内核里一气呵成,中间不会被打断。

```c
open("file", O_WRONLY | O_APPEND);
write(fd, buf, n);   // 原子操作,无竞争
```

适用:**追加写**(日志、传感器数据流)。

#### 解法 2:pread / pwrite(指定偏移读写)

```c
#include <unistd.h>
ssize_t pread(int fd, void *buf, size_t count, off_t offset);
ssize_t pwrite(int fd, const void *buf, size_t count, off_t offset);
```

**特点**:
- 把"lseek + read/write"合并成原子操作
- **不修改文件表里的当前偏移量**(线程友好)
- 比 lseek + read/write 性能好(少一次 syscall)

```c
// 不影响当前偏移量
pread(fd, buf, 1024, 500);   // 从偏移 500 读 1024 字节
```

适用:**多线程同一 fd 不同偏移读写**(数据库底层就靠这个)。

#### 解法 3:O_CREAT | O_EXCL(原子创建)

```c
int fd = open("lockfile", O_WRONLY | O_CREAT | O_EXCL, 0644);
if (fd == -1) {
    if (errno == EEXIST) {
        /* 文件已存在,说明别人持有锁 */
    }
}
```

**特点**:**"判断文件是否存在 + 创建"组合成原子操作**。如果文件已存在,返回 -1 + errno=EEXIST;否则创建。

适用:**互斥锁文件**、**单例进程检测**(Pidfile 机制)。

### 总结表

| 解法 | 解决什么场景 | 必备 flag |
|---|---|---|
| O_APPEND | 多进程追加写 | O_APPEND |
| pread/pwrite | 多线程随机读写 | 无 |
| O_CREAT|O_EXCL | 原子创建文件 | O_CREAT \| O_EXCL |

### 我的延伸:为什么"原子"在嵌入式驱动里也很重要

后面学**驱动并发与竞争**(驱动书第 47-48 章),内核里也有同样的问题,但工具不同:
- 用户态:O_APPEND / pwrite / 锁文件
- 内核态:自旋锁(spinlock)、互斥锁(mutex)、原子变量(atomic_t)、信号量(semaphore)

**思想完全一致:把多步操作合并为不可分割的整体**。本节是后面驱动学习的"前置课"。

## 3.10 fcntl 和 ioctl

### 3.10.1 fcntl —— 文件描述符瑞士军刀

```c
#include <fcntl.h>
int fcntl(int fd, int cmd, ... /* arg */);
```

#### cmd 的五大功能

| cmd | 作用 |
|---|---|
| `F_DUPFD` / `F_DUPFD_CLOEXEC` | 复制 fd(同 dup) |
| `F_GETFD` / `F_SETFD` | 获取/设置 fd 标志(主要是 close-on-exec) |
| `F_GETFL` / `F_SETFL` | 获取/设置文件状态标志(O_APPEND / O_NONBLOCK 等) |
| `F_GETOWN` / `F_SETOWN` | 异步 IO 信号接收者 |
| `F_GETLK` / `F_SETLK` | 文件记录锁(高级特性) |

#### 用法 1:复制 fd

```c
int fd2 = fcntl(fd1, F_DUPFD, 0);   // 等价 dup(fd1)
```

#### 用法 2:动态修改 flags(关键)

```c
// 把已打开的 fd 加上 O_APPEND
int flags = fcntl(fd, F_GETFL);
fcntl(fd, F_SETFL, flags | O_APPEND);
```

**典型用途**:把阻塞 IO 改成非阻塞:

```c
int flags = fcntl(fd, F_GETFL);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);
```

**限制**:`O_RDONLY` / `O_WRONLY` / `O_RDWR` / `O_CREAT` / `O_TRUNC` 不能用 F_SETFL 改,只有 `O_APPEND` / `O_ASYNC` / `O_DIRECT` / `O_NOATIME` / `O_NONBLOCK` 可以。

### 3.10.2 ioctl —— 驱动开发的接口

```c
#include <sys/ioctl.h>
int ioctl(int fd, unsigned long request, ...);
```

#### 这个函数极其重要 ⭐⭐⭐⭐⭐

- ioctl 是**驱动开发对外暴露自定义功能**的接口
- `fcntl` 处理标准化的 fd 操作
- `ioctl` 处理**设备特定的、非标准化的**操作
- **后面学驱动开发,自己写的字符设备驱动里 80% 要实现 ioctl**

#### 典型场景

```c
// 串口设置波特率
ioctl(uart_fd, TIOCSBRK, ...);

// 摄像头取一帧
ioctl(cam_fd, VIDIOC_DQBUF, &buf);

// LCD 获取屏幕参数
ioctl(lcd_fd, FBIOGET_VSCREENINFO, &info);

// 你自己写的传感器驱动
ioctl(sensor_fd, MY_SET_SAMPLE_RATE, 100);
```

#### 我的延伸:驱动里 ioctl 的实现

后面学第 40 章字符设备驱动时,你会写 `struct file_operations` 结构体,其中有个 `.unlocked_ioctl` 函数指针,用户态 ioctl 最终调用的就是这里。**整本应用书学到这里,等于已经为驱动学习开门了**。

## 3.11 截断文件 truncate / ftruncate

```c
#include <unistd.h>
int truncate(const char *path, off_t length);
int ftruncate(int fd, off_t length);
```

### 区别

- `truncate`:用**路径**指定文件(不需要 open)
- `ftruncate`:用 **fd** 指定文件(需要 open,且**可写权限**)

### 行为

- `length` **小于**当前大小 → 文件被砍掉,多余数据丢失
- `length` **大于**当前大小 → 文件被扩展,扩展部分全是 `\0`(变成空洞!)

### 使用陷阱

调用 `ftruncate` 后**不会修改当前偏移量**。如果偏移量原来指向被砍掉的位置,后续 IO 会出错。**用之前最好显式 lseek 一下**。

### 用法示例

```c
// 清空文件
int fd = open("file", O_RDWR);
ftruncate(fd, 0);

// 预分配 100MB 文件(创建空洞)
ftruncate(fd, 100 * 1024 * 1024);
```

## 章节深度小结

### 这一章在整本书里的地位

第 2 章教你"会用 IO",第 3 章教你"理解 IO"。**核心是建立"用户态 ↔ 内核态" 的心智模型**。

### 9 个必须记牢的知识点

1. **inode 不存文件名**,文件名存在目录项里
2. **PCB → fd 表 → 文件表 → inode**,**四级结构**是后面所有 IO 概念的基础
3. **open 后操作的是内存动态文件,不是磁盘**
4. **errno 是线程局部变量**,失败后立即检查,不要拖
5. **exit 刷缓冲,_exit 不刷**(printf 后用 _exit 可能丢输出)
6. **空洞文件物理大小 < 逻辑大小**,`ls` 显示逻辑,`du` 显示物理
7. **O_APPEND 是原子的,O_CREAT|O_EXCL 是原子的**
8. **dup/dup2 共享文件表,多次 open 是独立文件表**
9. **ioctl 是驱动开发对外的核心接口**,本章最重要的 syscall

#### 模板 1:严谨的 open + 错误处理

```c
int fd = open("file", O_RDONLY);
if (fd == -1) {

        fprintf(stderr, "File not found\n");

int fd = open("out.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);

```

int fd = open("app.log", O_WRONLY | O_CREAT | O_APPEND, 0644);

```

int fd = open("/var/run/myapp.pid", O_WRONLY | O_CREAT | O_EXCL, 0644);

```

## 踩坑追加(在原 13 条基础上新增)

### 14. _exit / exit 的输出丢失

`printf` 后跟 `_exit(0)`,输出可能丢(stdio 缓冲未刷)。改用 `exit(0)` 或 `printf` 后 `fflush(stdout)`。

### 15. perror 在成功路径用

我已经踩过 1 次,这里再强调:**perror 只在 `if (xxx == -1)` 分支里用**。

### 16. multiple open vs dup 的偏移行为差异

- 多次 open → **偏移独立**,容易竞争
- dup/dup2 → **偏移共享**,自然接续

如果你想"两个 fd 接续写",**用 dup 不要 open 两次**。

### 17. O_APPEND 改写偏移无效

```c
fd = open("file", O_RDWR | O_APPEND);
lseek(fd, 0, SEEK_SET);   // 看似把偏移调到开头
write(fd, "X", 1);         // 实际写到末尾!O_APPEND 强制末尾
```

### 18. truncate 后忘了 lseek

```c
ftruncate(fd, 100);         // 文件砍到 100 字节
// 当前偏移可能 > 100
write(fd, buf, 10);         // 在 > 100 处写,产生空洞
```

需要 `lseek(fd, 0, SEEK_SET)` 重新定位。
