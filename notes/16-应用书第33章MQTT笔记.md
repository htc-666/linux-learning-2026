## 一、MQTT 是什么
```
轻量级【发布/订阅(pub/sub)】消息协议,为物联网/低带宽/弱网设计
三角色: Broker(服务端/代理,消息中转枢纽) / Publisher(发布者) / Subscriber(订阅者)
Topic(主题): 消息的"频道",发布订阅都围绕topic;发布者订阅者只认topic,互不认识
```
### 发布/订阅三特性(都带"可"字=灵活)
```
客户端相互独立 + 空间上可分离 + 时间上可异步(订阅者不在线,消息可保存待其上线)
```
### 为什么物联网用MQTT(面试)
```
轻量(头部小,省流量省电) + 长连接 + 发布订阅解耦 + QoS可靠 → 适合海量设备/弱网/省电
vs HTTP: MQTT长连接+推送+省流量;HTTP请求响应+重,不适合持续上报
```

## 二、连接 CONNECT / CONNACK
```
客户端→服务端: CONNECT报文(请求连接)
服务端→客户端: CONNACK报文(确认, returnCode=0成功)
CONNECT关键字段:
  clientID    客户端唯一标识(不能重复)
  keepAlive   心跳间隔(秒),如60
  cleanSession 清除会话:
     =1 新会话,断开就清空(不保存订阅/离线消息)
     =0 持久会话,服务端保存订阅的topic+离线消息,重连后补发  ← 想收离线消息(QoS>0)必须=0!
  username/password 可选,认证用
  will... 遗嘱(见下)
```

## 三、发布 / 订阅 / 取消订阅
```
PUBLISH(发布):   发到某topic; 字段 topicName/payload(内容)/qos/retain/dup
SUBSCRIBE(订阅): 订阅topic; 可一次订多个; 服务端回 SUBACK
UNSUBSCRIBE:    取消订阅; 服务端回 UNSUBACK
```

## 四、主题 Topic
```
分级: home/sensor/led/brightness  (用 / 分级)
通配符(只能订阅时用):
  + 单级: home/sensor/+/status  匹配 home/sensor/led/status 等(中间一级任意)
  # 多级: home/sensor/#         匹配 home/sensor 下所有层级
$开头: $SYS/... 是系统保留主题,别乱用
规范: 别用/开头、别用空格、用ASCII、保持简洁、区分大小写
```

## 五、QoS 服务质量等级 ⭐⭐⭐(面试核心)
```
QoS=0 最多一次: 发完不管,可能丢(靠TCP,不额外确认)        → 不重要数据
QoS=1 至少一次: 收到回PUBACK确认;没收到就重发 → 可能重复  → 一般用这个
QoS=2 恰好一次: 四步握手 PUBLISH→PUBREC→PUBREL→PUBCOMP    → 最可靠最慢,不重复
设置: PUBLISH和SUBSCRIBE报文里都有qos字段
降级: 发布QoS2但订阅者只订QoS1 → 服务端按【较低】的QoS1发给他
注意: QoS1可能重复,去重要靠你应用程序(协议不去重),看dup标志
```

## 六、保留消息 retain ⭐
```
retain=true: 服务端【保留】该topic最后一条消息
  → 新订阅者一订阅,立刻收到这条保留消息(不用等下次发布)
适用: "状态类"数据(如室温每小时才发一次,新订阅者想立刻知道当前温度)
更新: 发新的保留消息覆盖旧的; 删除: 发一条空的保留消息
```

## 七、心跳机制 keepAlive ⭐
```
客户端空闲时,定时发心跳包 PINGREQ → 服务端回 PINGRESP
作用: 服务端判断客户端是否在线(长时间没心跳=掉线)
间隔由 CONNECT 的 keepAlive 指定(如60秒)
```

## 八、遗嘱机制 LWT(Last Will and Testament)⭐⭐(面试亮点)
```
客户端【意外掉线】(没主动DISCONNECT,如断电/断网)时,
服务端【自动发布】客户端预设的"遗嘱"消息给订阅者 → 通知"某设备离线了"
设置: CONNECT报文里 willTopic/willMessage/willRetain/willQos
关键: 只有【意外断线】才触发遗嘱;主动DISCONNECT不触发
用途: 设备掉线告警(如"传感器A离线")
```

## 九、用户密码认证
```
CONNECT的 username/password: 服务端可要求认证,认证通过才能连
也可用于区分用户/私有主题权限
```

## 🔵 对口你的毕设(OneNET)
```
你毕设: STM32→AT指令→ESP8266→MQTT→OneNET(broker)
  publish 风速数据到 OneNET 的数据点(topic),OneJSON格式,Token认证
对照本章:
  broker=OneNET; publisher=你的设备; topic=数据点; 认证=Token(类似username/password)
能讲: "设备用MQTT publish数据到OneNET,选MQTT因为轻量省电(我做低功耗设计)"
```

## 一句话总结
> **MQTT=物联网轻量发布订阅协议**:broker中转,客户端围绕topic发布(PUBLISH)/订阅(SUBSCRIBE)。
> **核心机制:QoS(0/1/2可靠等级)、保留消息(新订阅者立刻收)、心跳(判在线)、遗嘱LWT(掉线告警)。**
> **你毕设的OneNET就是个broker,理解这些就能讲清你的MQTT上云。代码(Linux库)跳。**
