## 本章总览

第40章的核心不是某个具体硬件，而是把 **Linux 字符设备驱动的基本框架** 跑通。

用户态程序看到的是：

```text
/dev/chrdevbase
```

应用程序对它调用：

```c
open()
read()
write()
close()
```

内核里真正响应的是驱动注册进去的：

```c
chrdevbase_open()
chrdevbase_read()
chrdevbase_write()
chrdevbase_release()
```

这条映射关系就是本章的灵魂：

```text
应用层 open/read/write/close
        ↓ C库 + 系统调用
内核 VFS
        ↓ 根据设备号找到字符设备驱动
驱动 file_operations 里的 open/read/write/release
        ↓
具体设备逻辑
```

🔵 **你现在要建立的直觉**：Linux 驱动不是应用层直接调用你写的函数，而是你把一组函数注册给内核，应用层通过 `/dev/xxx` 这个设备文件间接触发它们。

## 40.1 字符设备驱动简介 ⭐⭐⭐⭐⭐

### 1. 字符设备是什么

字符设备就是按照 **字节流** 顺序读写的设备。常见例子：

| 设备 | 为什么是字符设备 |
|---|---|
| LED | 应用层写入控制命令，驱动按字节接收 |
| 按键 | 应用层读取按键状态 |
| 串口 | 数据天然按字节流收发 |
| I2C / SPI 设备 | 通常通过 read/write/ioctl 等接口访问 |
| LCD | 有些接口也可体现为字符设备或 framebuffer 设备 |

字符设备和块设备的直观区别：

| 类型 | 访问方式 | 例子 |
|---|---|---|
| 字符设备 | 按字节流顺序访问 | 串口、LED、按键、I2C 传感器 |
| 块设备 | 按块访问，适合随机读写 | eMMC、SD 卡、U 盘、硬盘 |

🔵 **面试常问**：字符设备和块设备区别是什么？  
答的时候不要只说“一个字节一个块”，要补一句：字符设备通常不经过块缓存层，常用 `file_operations` 实现 open/read/write/ioctl；块设备面向存储，有请求队列、调度、缓存等机制。

### 2. 应用程序如何调用驱动

Linux 里“一切皆文件”。驱动加载成功后，用户通常会在 `/dev` 下看到一个设备节点，例如：

```text
/dev/led
/dev/chrdevbase
```

应用层像操作普通文件一样操作它：

```c
fd = open("/dev/chrdevbase", O_RDWR);
read(fd, buf, len);
write(fd, buf, len);
close(fd);
```

但这些函数不是直接进入你的驱动源码，而是先进入 C 库和系统调用，再由内核 VFS 找到对应驱动。

```text
open()
  → C库 open()
  → open 系统调用
  → VFS
  → 驱动 file_operations.open
```

### 3. file_operations：驱动的函数表

`struct file_operations` 定义在内核源码 `include/linux/fs.h`，它是字符设备驱动最核心的结构体。

常用成员：

| 成员 | 对应应用层接口 | 作用 |
|---|---|---|
| `.owner` | 无直接对应 | 一般填 `THIS_MODULE`，表示本模块拥有该操作集 |
| `.open` | `open()` | 打开设备 |
| `.release` | `close()` | 关闭/释放设备 |
| `.read` | `read()` | 从设备读数据到用户空间 |
| `.write` | `write()` | 从用户空间写数据到设备 |
| `.unlocked_ioctl` | `ioctl()` | 执行控制类命令 |
| `.poll` | `poll()/select()` | 支持非阻塞/阻塞等待事件 |
| `.mmap` | `mmap()` | 把设备内存映射到用户空间 |

本章最小闭环只实现：

```c
static struct file_operations chrdevbase_fops = {
    .owner = THIS_MODULE,
    .open = chrdevbase_open,
    .read = chrdevbase_read,
    .write = chrdevbase_write,
    .release = chrdevbase_release,
};
```

🔵 **你的理解锚点**：`file_operations` 就像“驱动的函数菜单”。应用层点了 `read()` 这道菜，内核就去菜单里找 `.read` 对应的函数。

## 40.2 字符设备驱动开发步骤 ⭐⭐⭐⭐⭐

这一节给出字符设备驱动的标准老框架。

### 40.2.1 驱动模块的加载和卸载 ⭐⭐⭐⭐⭐

Linux 驱动有两种运行方式：

| 方式 | 特点 | 适合场景 |
|---|---|---|
| 编译进内核 | 随内核启动自动运行 | 稳定、量产、基础驱动 |
| 编译成模块 `.ko` | 系统启动后用 `insmod/modprobe` 加载 | 开发调试阶段 |

本章采用模块方式，因为调试方便：

```text
改驱动源码
  → make 编译出 .ko
  → insmod 加载
  → rmmod 卸载
```

模块入口/出口模板：

```c
static int __init xxx_init(void)
{
    return 0;
}

static void __exit xxx_exit(void)
{
}

module_init(xxx_init);
module_exit(xxx_exit);
```

关键点：

| 写法 | 含义 |
|---|---|
| `__init` | 入口函数用，加载后这部分内存可释放 |
| `__exit` | 出口函数用，卸载模块时调用 |
| `module_init()` | 注册模块加载函数，`insmod` 时触发 |
| `module_exit()` | 注册模块卸载函数，`rmmod` 时触发 |

加载/卸载命令：

```bash
insmod chrdevbase.ko
rmmod chrdevbase.ko
```

`insmod` 和 `modprobe` 的区别：

| 命令 | 特点 |
|---|---|
| `insmod xxx.ko` | 直接加载指定模块，不自动处理依赖 |
| `modprobe xxx` | 会查 `/lib/modules/<kernel-version>/`，能处理依赖 |
| `rmmod xxx.ko` | 卸载指定模块 |
| `modprobe -r xxx` | 卸载模块并处理依赖 |

🔵 **你的板子场景要特别记住**：正点原子内核是 `4.1.15`，模块要和板子正在运行的内核版本一致，否则会遇到 `vermagic` 不匹配，`insmod` 失败。

### 40.2.2 字符设备注册与注销 ⭐⭐⭐⭐⭐

老字符设备注册 API：

```c
static inline int register_chrdev(unsigned int major,
                                  const char *name,
                                  const struct file_operations *fops);

static inline void unregister_chrdev(unsigned int major,
                                     const char *name);
```

参数含义：

| 参数 | 含义 |
|---|---|
| `major` | 主设备号 |
| `name` | 设备名，会出现在 `/proc/devices` |
| `fops` | 设备操作函数集合 |

典型写法：

```c
static struct file_operations test_fops;

static int __init xxx_init(void)
{
    int retvalue = 0;

    retvalue = register_chrdev(200, "chrtest", &test_fops);
    if (retvalue < 0) {
        /* 注册失败处理 */
    }

    return 0;
}

static void __exit xxx_exit(void)
{
    unregister_chrdev(200, "chrtest");
}
```

检查主设备号是否已被占用：

```bash
cat /proc/devices
```

🔵 **易错点**：

- `register_chrdev()` 成功注册的是字符设备驱动，不等于 `/dev/xxx` 节点已经自动存在。
- 本章用的是老 API，所以后面还要手动 `mknod` 创建设备节点。
- 主设备号随便写容易冲突，所以第42章会学动态申请设备号和 `cdev`。

### 40.2.3 实现设备的具体操作函数 ⭐⭐⭐⭐⭐

本章最小需求：

1. 能打开和关闭 chrtest/chrdevbase 设备。
2. 能从设备读数据。
3. 能向设备写数据。

对应实现：

```c
static int chrtest_open(struct inode *inode, struct file *filp)
{
    return 0;
}

static ssize_t chrtest_read(struct file *filp,
                            char __user *buf,
                            size_t cnt,
                            loff_t *offt)
{
    return 0;
}

static ssize_t chrtest_write(struct file *filp,
                             const char __user *buf,
                             size_t cnt,
                             loff_t *offt)
{
    return 0;
}

static int chrtest_release(struct inode *inode, struct file *filp)
{
    return 0;
}
```

然后绑定进 `file_operations`：

```c
static struct file_operations test_fops = {
    .owner = THIS_MODULE,
    .open = chrtest_open,
    .read = chrtest_read,
    .write = chrtest_write,
    .release = chrtest_release,
};
```

几个参数要认识：

| 参数 | 出现位置 | 作用 |
|---|---|---|
| `struct inode *inode` | open/release | 保存设备号等 inode 信息 |
| `struct file *filp` | open/read/write/release | 表示一次打开的文件对象 |
| `char __user *buf` | read | 用户空间缓冲区地址 |
| `const char __user *buf` | write | 用户空间传来的数据地址 |
| `size_t cnt` | read/write | 请求读写的字节数 |
| `loff_t *offt` | read/write | 文件偏移 |

🔵 `__user` 的意义：它提醒开发者和静态检查工具，这个指针来自用户空间，内核不能像普通指针一样直接解引用。

### 40.2.4 添加 LICENSE 和作者信息 ⭐⭐⭐

模块末尾必须添加 LICENSE：

```c
MODULE_LICENSE("GPL");
MODULE_AUTHOR("zuozhongkai");
```

`MODULE_LICENSE("GPL")` 很重要。没有 LICENSE 时，内核会认为模块污染内核，编译/加载时可能报警。

## 40.3 Linux 设备号 ⭐⭐⭐⭐⭐

### 40.3.1 设备号的组成 ⭐⭐⭐⭐⭐

Linux 用设备号管理设备。

设备号由两部分组成：

| 部分 | 含义 |
|---|---|
| 主设备号 major | 表示某一类/某一个驱动 |
| 次设备号 minor | 表示该驱动下的具体设备 |

例如：

```text
/dev/chrdevbase c 200 0
```

含义是：

| 字段 | 含义 |
|---|---|
| `c` | character，字符设备 |
| `200` | 主设备号 |
| `0` | 次设备号 |

内核中用 `dev_t` 表示设备号，本质是 32 位无符号数：

```c
typedef __kernel_dev_t dev_t;
typedef unsigned int __u32;
```

在本书内核里：

| 位段 | 含义 |
|---|---|
| 高 12 位 | 主设备号 |
| 低 20 位 | 次设备号 |

设备号相关宏：

```c
#define MINORBITS 20
#define MINORMASK ((1U << MINORBITS) - 1)

#define MAJOR(dev) ((unsigned int) ((dev) >> MINORBITS))
#define MINOR(dev) ((unsigned int) ((dev) & MINORMASK))
#define MKDEV(ma, mi) (((ma) << MINORBITS) | (mi))
```

常用含义：

| 宏 | 作用 |
|---|---|
| `MAJOR(dev)` | 从 `dev_t` 中取主设备号 |
| `MINOR(dev)` | 从 `dev_t` 中取次设备号 |
| `MKDEV(major, minor)` | 把主设备号和次设备号合成 `dev_t` |

🔵 **面试小坑**：主设备号不是无限的。本章讲的是高 12 位，所以范围是 `0~4095`。

### 40.3.2 设备号的分配 ⭐⭐⭐⭐⭐

设备号有两种分配方式。

#### 1. 静态分配

开发者自己选一个主设备号，比如本章：

```c
#define CHRDEVBASE_MAJOR 200
```

优点：简单，适合入门实验。

缺点：

- 可能和系统已有设备冲突。
- 移植到不同板子时不一定安全。
- 需要先用 `cat /proc/devices` 检查。

#### 2. 动态分配

社区更推荐动态申请：

```c
int alloc_chrdev_region(dev_t *dev,
                        unsigned baseminor,
                        unsigned count,
                        const char *name);

void unregister_chrdev_region(dev_t from, unsigned count);
```

参数含义：

| 参数 | 含义 |
|---|---|
| `dev` | 保存申请到的设备号 |
| `baseminor` | 起始次设备号，一般填 0 |
| `count` | 要申请的设备号数量 |
| `name` | 设备名 |

🔵 **第40章 vs 第42章**：

| 第40章 | 第42章 |
|---|---|
| `register_chrdev(200, ...)` | `alloc_chrdev_region()` / `register_chrdev_region()` |
| 手动指定主设备号 | 动态/规范申请设备号 |
| 手动 `mknod` | 可通过 class/device 自动创建设备节点 |
| 入门简单 | 更符合真实驱动写法 |

## 40.4 chrdevbase 字符设备驱动开发实验 ⭐⭐⭐⭐⭐

本节用一个虚拟设备 `chrdevbase` 跑通第一个字符设备驱动。

它不操作真实硬件，只维护两个内核缓冲区：

```c
static char readbuf[100];
static char writebuf[100];
static char kerneldata[] = {"kernel data!"};
```

实验目标：

| 操作 | 现象 |
|---|---|
| 应用层读 `/dev/chrdevbase` | 驱动把 `"kernel data!"` 拷到用户空间 |
| 应用层写 `/dev/chrdevbase` | 驱动收到 `"usr data!"` 并用 `printk` 打印 |

### 40.4.1 实验程序编写 ⭐⭐⭐⭐⭐

#### 1. 工程结构

书中路径：

```text
Linux_Drivers/
└── 1_chrdevbase/
    ├── chrdevbase.c
    ├── chrdevbaseApp.c
    └── Makefile
```

你当前建议放在 Ubuntu：

```bash
mkdir -p ~/linux-learning/ch40/1_chrdevbase
cd ~/linux-learning/ch40/1_chrdevbase
```

#### 2. VSCode 头文件路径

驱动会引用内核头文件，所以 VSCode 要加内核源码头文件路径。

按你现在的环境，`KERNELDIR` 应该是：

```text
~/linux/alientek_kernel
```

可以在 `.vscode/c_cpp_properties.json` 里加：

```json
"includePath": [
    "${workspaceFolder}/**",
    "/home/book/linux/alientek_kernel/include",
    "/home/book/linux/alientek_kernel/arch/arm/include",
    "/home/book/linux/alientek_kernel/arch/arm/include/generated"
]
```

🔵 这里的 `/home/book/` 按你虚拟机用户 `book@100ask` 写；如果你的实际路径不同，以 `pwd` 为准。

#### 3. chrdevbase.c 的核心结构

这一章的驱动源码可以拆成 7 块：

```text
头文件
  ↓
宏定义：主设备号、设备名
  ↓
内核缓冲区
  ↓
open/read/write/release
  ↓
file_operations
  ↓
init/exit 中注册/注销字符设备
  ↓
MODULE_LICENSE / MODULE_AUTHOR
```

核心宏：

```c
#define CHRDEVBASE_MAJOR 200
#define CHRDEVBASE_NAME  "chrdevbase"
```

读函数核心逻辑：

```c
memcpy(readbuf, kerneldata, sizeof(kerneldata));
retvalue = copy_to_user(buf, readbuf, cnt);
```

写函数核心逻辑：

```c
retvalue = copy_from_user(writebuf, buf, cnt);
printk("kernel recevdata:%s\r\n", writebuf);
```

`copy_to_user`：

```c
static inline long copy_to_user(void __user *to,
                                const void *from,
                                unsigned long n);
```

`copy_from_user`：

```c
static inline long copy_from_user(void *to,
                                  const void __user *from,
                                  unsigned long n);
```

| 函数 | 方向 | 记忆法 |
|---|---|---|
| `copy_to_user(to, from, n)` | 内核 → 用户 | read 时用，把内核数据给用户 |
| `copy_from_user(to, from, n)` | 用户 → 内核 | write 时用，把用户数据拿进内核 |

🔵 **非常容易写反**：  
应用层 `read()` 对应驱动里 `copy_to_user()`；应用层 `write()` 对应驱动里 `copy_from_user()`。

#### 4. printk 不是 printf

内核里没有 `printf()`，要用：

```c
printk("chrdevbase_init()\r\n");
```

`printk` 支持日志级别：

| 级别 | 含义 |
|---|---|
| `KERN_EMERG` | 紧急，内核崩溃级 |
| `KERN_ALERT` | 必须立即处理 |
| `KERN_CRIT` | 严重错误 |
| `KERN_ERR` | 错误 |
| `KERN_WARNING` | 警告 |
| `KERN_NOTICE` | 普通但值得注意 |
| `KERN_INFO` | 信息 |
| `KERN_DEBUG` | 调试 |

例子：

```c
printk(KERN_INFO "chrdevbase init\r\n");
```

🔵 入门实验里不写日志级别也能显示，因为默认级别通常足够高；真实驱动里建议带 `KERN_ERR/KERN_INFO`。

### 40.4.2 编写测试 APP ⭐⭐⭐⭐

测试 APP 是用户态程序，不是驱动。

它调用：

```c
open()
read()
write()
close()
```

用法：

```bash
./chrdevbaseApp /dev/chrdevbase 1
./chrdevbaseApp /dev/chrdevbase 2
```

参数含义：

| 参数 | 含义 |
|---|---|
| `argv[0]` | 程序名 `./chrdevbaseApp` |
| `argv[1]` | 设备节点 `/dev/chrdevbase` |
| `argv[2]` | 操作类型：`1` 读，`2` 写 |

核心逻辑：

```c
fd = open(filename, O_RDWR);

if (atoi(argv[2]) == 1) {
    retvalue = read(fd, readbuf, 50);
    printf("read data:%s\r\n", readbuf);
}

if (atoi(argv[2]) == 2) {
    memcpy(writebuf, usrdata, sizeof(usrdata));
    retvalue = write(fd, writebuf, 50);
}

close(fd);
```

🔵 **这里特别能体现“一切皆文件”**：应用层根本不知道下面有没有真实硬件，只要有 `/dev/chrdevbase`，它就按普通文件接口操作。

### 40.4.4 运行测试 ⭐⭐⭐⭐⭐


具体要看你当时导出的 rootfs 目录。如果是直接导出 `~/linux/nfs`，可以先放：

```bash
cp chrdevbase.ko chrdevbaseApp ~/linux/nfs/
```

如果你的板子根文件系统里有：

```text
/lib/modules/4.1.15/
```

也可以放到对应目录，方便 `modprobe` 查找。

#### 2. 加载驱动

在开发板上执行：

```bash
insmod chrdevbase.ko
```

查看模块：

```bash
lsmod
```

查看设备是否注册：

```bash
cat /proc/devices
```

应该能看到类似：

```text
200 chrdevbase
```

如果使用：

```bash
modprobe chrdevbase.ko
```

可能遇到 `modules.dep` 不存在，这时执行：

```bash
depmod
```

但入门阶段直接用 `insmod` 更直观。

#### 3. 创建设备节点

第40章不会自动创建 `/dev/chrdevbase`，需要手动：

```bash
mknod /dev/chrdevbase c 200 0
```

查看：

```bash
ls -l /dev/chrdevbase
```

含义：

| 字段 | 含义 |
|---|---|
| `mknod` | 创建设备节点 |
| `/dev/chrdevbase` | 节点路径 |
| `c` | 字符设备 |
| `200` | 主设备号 |
| `0` | 次设备号 |

🔵 **关键理解**：`register_chrdev()` 是把驱动注册进内核；`mknod` 是在 `/dev` 下创建用户态能打开的入口。两者缺一不可。

#### 4. 读写测试

读测试：

```bash
./chrdevbaseApp /dev/chrdevbase 1
```

期望现象：

```text
kernel senddata ok!
read data:kernel data!
```

写测试：

```bash
./chrdevbaseApp /dev/chrdevbase 2
```

期望现象：

```text
kernel recevdata:usr data!
```

如果终端没看到内核日志，可以看：

```bash
dmesg
```

#### 5. 卸载驱动

```bash
rmmod chrdevbase.ko
lsmod
```

正常情况下，`chrdevbase` 不再出现在 `lsmod` 中。

## 易错点清单

| 易错点 | 现象 | 原因/解决 |
|---|---|---|
| 没有创建设备节点 | `Can't open file /dev/chrdevbase` | `insmod` 后还要 `mknod /dev/chrdevbase c 200 0` |
| 主设备号冲突 | `register failed` 或 `/proc/devices` 已有 200 | 换主设备号，或后面用动态申请 |
| 用本机 gcc 编译 APP | 板子运行报格式错误 | APP 要用 `arm-linux-gnueabihf-gcc` |
| `.ko` 内核版本不匹配 | `invalid module format` | 驱动必须用板子正在运行的内核源码编译 |
| `copy_to_user/from_user` 写反 | 读写逻辑异常 | read 用 `copy_to_user`，write 用 `copy_from_user` |
| 内核里用 `printf` | 编译失败 | 内核打印用 `printk` |
| 忘记 `MODULE_LICENSE("GPL")` | 编译/加载警告 | 模块末尾加 LICENSE |
| Makefile 路径错误 | 找不到内核构建目录 | `KERNELDIR` 改成 `/home/book/linux/alientek_kernel` |
| `read/write` 返回值一直 0 | APP 不一定能感知真实读写字节数 | 入门例程简化了，真实驱动应返回实际读写字节数 |

🔵 最后一条尤其重要：书上 `chrdevbase_read/write` 最后 `return 0;` 是为了讲框架简化。真实驱动里 `read()`/`write()` 应该返回实际成功传输的字节数，失败返回负错误码。

## 面试速查

**1. 字符设备驱动最核心的结构体是什么？**  
`struct file_operations`。它保存驱动对外提供的操作函数，如 `.open/.read/.write/.release/.unlocked_ioctl`。

**2. 应用层 `read()` 最后怎么调用到驱动？**  
应用层 `read()` 进入 C 库和系统调用，内核 VFS 根据文件描述符找到对应设备文件，再根据设备号找到字符设备驱动，最终调用 `file_operations.read`。

**3. 主设备号和次设备号分别表示什么？**  
主设备号通常表示一个驱动，次设备号表示这个驱动管理的某个具体设备。

**4. `register_chrdev()` 做了什么？**  
把一个字符设备驱动注册进内核，并把主设备号、设备名、`file_operations` 绑定起来。

**5. `register_chrdev()` 之后为什么还要 `mknod`？**  
注册只是内核知道了这个驱动；`mknod` 是创建 `/dev/xxx` 设备节点，让用户态程序能打开它。

**6. `copy_to_user()` 和 `copy_from_user()` 为什么必须用？**  
用户空间和内核空间地址隔离，内核不能直接访问用户指针，必须通过这两个函数安全拷贝。

**7. read/write 里 `buf` 是用户空间还是内核空间？**  
`read` 的 `char __user *buf` 和 `write` 的 `const char __user *buf` 都是用户空间地址。

**8. `insmod` 和 `modprobe` 的区别？**  
`insmod` 直接加载指定 `.ko`，不处理依赖；`modprobe` 会从 `/lib/modules/<kernel-version>` 查找并处理依赖。

**9. 为什么驱动模块必须和内核版本匹配？**  
模块依赖内核符号、配置和 vermagic。用 4.1.15 编出来的 `.ko` 不能随便加载到 4.9.88 内核。

**10. 第40章老字符设备框架有什么缺点？**  
静态主设备号容易冲突，手动 `mknod` 麻烦，`register_chrdev` 一次占用整个主设备号下的次设备号范围。第42章会用 `cdev`、动态设备号、class/device 改进。

## 一句话总结

第40章不是为了学一个虚拟设备，而是让你打通字符设备驱动的第一条完整链路：**应用层文件 IO → `/dev` 设备节点 → 设备号 → `file_operations` → 内核驱动函数 → 用户/内核数据拷贝**。
