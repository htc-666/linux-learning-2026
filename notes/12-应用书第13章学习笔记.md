## 13.1 非阻塞I/O ⭐⭐⭐(多路复用的铺垫)

```
阻塞I/O:   无数据可读 → 进程【睡眠】,让出CPU,等数据来才醒
非阻塞I/O: 无数据可读 → 【立刻返回错误】(EAGAIN/EWOULDBLOCK),不等
```
- 开启方式:`open(..., O_NONBLOCK)` 或 `fcntl(fd, F_SETFL, flag | O_NONBLOCK)`
- 普通文件读写不阻塞(总有结果);**管道/设备/socket** 才有阻塞概念

**优缺点对比**:
```
阻塞   : 省CPU(睡着等),但一个线程同时只能等一个fd → 无法并发读多个设备
非阻塞 : 能轮询多个fd(并发),但要【死循环不停试】→ CPU占用极高(top能看到~100%)
→ 两者都不完美 → 引出 13.2 多路复用(既并发又不烧CPU)
```

## 13.2 I/O多路复用 select/poll ⭐⭐⭐⭐(本章核心)

**一句话**:一个线程"同时盯多个fd",哪个就绪处理哪个,期间睡着不烧CPU。
特征:**外部阻塞式,内部监视多路I/O**。

### select
```c
#include <sys/select.h>
int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);
// nfds = 最大fd + 1;  timeout 是 struct timeval*(可NULL一直等)
// 返回: >0 就绪fd数 / ==0 超时 / <0 出错

// 四个操作宏:
FD_ZERO(&set);        // 清空
FD_SET(fd, &set);     // 加入fd
FD_CLR(fd, &set);     // 移除fd
FD_ISSET(fd, &set);   // 查fd是否就绪
```

**三步套路**:
```c
FD_ZERO(&rfds); FD_SET(fd, &rfds);          // ①装fd
struct timeval tv = {5, 0};
ret = select(fd+1, &rfds, NULL, NULL, &tv); // ②阻塞等(带超时)
if (FD_ISSET(fd, &rfds)) { read(...); }     // ③查就绪+读
```

### 🔴 demo 1 踩坑实录 (ch13/01_select.c)
监听键盘(fd 0),5秒超时:
```
现象: 5秒内打字→读出字节数; 5秒不动→time out
踩的坑(都很典型):
  ① 超时参数写成整数 5 → 必须是 struct timeval*(struct timeval tv={5,0}; 传&tv)
  ② fd_set 要传地址 &fds(不是 fds)
  ③ printf("...字节"\n,s) → \n 跑到引号外了,要 "...字节\n"
  ④ ⭐放循环里时: tv 没重置 → 第1次等5秒,后面秒刷time out!
     根因: select 返回时会【改写 fd_set 和 tv】(tv被改成剩余时间)
     → 循环里每轮必须重新 FD_ZERO+FD_SET + 重设 tv
  ⑤ 用了select就不必再设O_NONBLOCK(那是轮询方式才要的)
```

### poll(select 的改良版)
```c
#include <poll.h>
int poll(struct pollfd *fds, nfds_t nfds, int timeout);  // timeout是【毫秒】整数(-1一直等)
struct pollfd {
    int   fd;
    short events;    // 关心的事件(输入): POLLIN(可读) / POLLOUT(可写)
    short revents;   // 实际发生的事件(输出,内核填)
};
// 用 fds[i].revents & POLLIN 判断是否可读
```
- poll vs select:**poll 用 pollfd 数组,没有 select 的 fd 上限(1024)**;用法几乎一样。
- 代码只需写 select 一个就够,poll 知道区别即可。

### 13.2.4 重要:就绪后必须执行I/O清除状态
select/poll 报告某 fd 就绪后,**必须真的去 read/write**,否则该状态一直存在,下次调用立即又返回。

### 🔵 epoll(面试重点!书没讲,必须懂)
```c
int epoll_create(int size);                          // 建epoll实例
int epoll_ctl(epfd, op, fd, struct epoll_event*);    // 增/删/改要监听的fd
int epoll_wait(epfd, events, maxevents, timeout);     // 等就绪,只返回就绪的fd
```

**select / poll / epoll 对比(面试必背)**:
| | select | poll | epoll |
|---|---|---|---|
| fd上限 | 有(1024) | 无 | 无 |
| 数据结构 | fd_set位图 | pollfd数组 | 内核红黑树+就绪链表 |
| 每次调用 | 拷贝全部fd+遍历 **O(n)** | 同select | **O(1)**,只返回就绪的 |
| 海量连接 | 慢 | 慢 | **快(高并发标配)** |

```
epoll 为什么高效(高频考点):
  ① 不用每次把所有fd从用户态拷到内核态(epoll_ctl注册一次,内核记着)
  ② 不用遍历所有fd → 内核用回调把就绪的fd放进就绪链表,epoll_wait只返回就绪的
  → 连接数上万时,远胜 select/poll 的 O(n) 轮询
触发模式: LT(水平触发,默认,没读完会再通知) / ET(边沿触发,只通知一次,要一次读干净)
```

## 13.3 异步I/O(信号驱动I/O) ⭐(了解)

```
原理: fd可操作时,内核【主动发信号】通知进程(默认 SIGIO),不用进程自己查
步骤: open加 O_NONBLOCK → fcntl加 O_ASYNC → fcntl(F_SETOWN,getpid()) 设接收进程
      → signal(SIGIO, handler) → 信号来时在 handler 里读
```
了解即可,实战少用。

## 13.4 优化异步I/O ⭐(了解)
```
SIGIO 的两个缺陷:
  ① SIGIO是【非排队信号】,处理慢会丢信号
  ② handler里不知道fd发生了什么事件
优化: 用【实时信号 SIGRTMIN】替换SIGIO(可排队) + sigaction(SA_SIGINFO)
      → 用 siginfo_t 的 si_code/si_fd 判断哪个fd、什么事件
注意: 用 F_SETSIG 要先 #define _GNU_SOURCE
```
了解即可。

## 13.5 存储映射 mmap ⭐⭐⭐⭐(面试+后面用+驱动核心)

**一句话**:把文件映射成内存,**读写内存 = 读写文件**,不用调 read/write。

```c
#include <sys/mman.h>
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
int   munmap(void *addr, size_t length);
// addr一般NULL(系统选); length映射长度; offset从哪开始(一般0)
// prot:  PROT_READ / PROT_WRITE / PROT_EXEC / PROT_NONE
// flags: MAP_SHARED(写回文件,可共享) / MAP_PRIVATE(写时复制,不回写) / MAP_ANONYMOUS(匿名,不关联文件)
// 失败返回 MAP_FAILED(不是-1!)
```

### 🔴 demo 2 踩坑实录 (ch13/02_mmap.c) —— mmap 实现文件复制
```c
srcfd = open(argv[1], O_RDONLY);                          // 源:只读
dstfd = open(argv[2], O_RDWR | O_CREAT | O_TRUNC, 0664);  // 目标:读写+创建+清空
fstat(srcfd, &sbuf);                                       // 取源大小
ftruncate(dstfd, sbuf.st_size);                            // 目标撑到同样大
src = mmap(NULL, sbuf.st_size, PROT_READ,  MAP_SHARED, srcfd, 0);
dst = mmap(NULL, sbuf.st_size, PROT_WRITE, MAP_SHARED, dstfd, 0);
memcpy(dst, src, sbuf.st_size);                            // 复制就这一行!
munmap(src,...); munmap(dst,...); close(srcfd); close(dstfd);
```
```
踩的坑(4个,都典型):
  ① 源文件误加 O_CREAT|O_TRUNC → 会清空源文件! 源只能 O_RDONLY
  ② 目标文件 O_WRONLY → mmap失败! ⭐MAP_SHARED+PROT_WRITE 要求 fd 必须 O_RDWR
  ③ memcpy 方向反(应 memcpy(dst,src)); 写只读映射区会 SIGSEGV
  ④ srcfd 笔误开成 argv[2]
关键: 目标文件必须先 ftruncate 撑大,否则写映射区触发 SIGBUS
```

### 两个相关信号
```
SIGSEGV: 往【只读】映射区写数据 → 段错误
SIGBUS:  访问映射区里【文件不存在】的部分(如目标没ftruncate撑大) → 总线错误
```

### 其它函数
```
msync(addr, len, MS_SYNC): 把映射区改动【刷回磁盘】(类比fsync,默认不会立即写盘)
mprotect(addr, len, prot): 改映射区的读写权限
```

### 普通I/O vs mmap(面试)
```
普通I/O(read/write): 数据要在"内核缓冲↔用户缓冲"来回拷 → 慢
mmap: 文件直接映射到进程地址空间,读写内存即读写文件 → 少一次拷贝(零拷贝思想)
适用: mmap适合【大数据/频繁访问】(视频图像); 少量数据用普通I/O更省事
```

## 13.6 文件锁 ⭐⭐(了解)

**作用**:多进程操作同一文件时,用锁避免数据冲突(类比互斥锁,但用于"进程间+文件")。

```
建议性锁: 大家约定"访问前先查锁",但不查直接访问也能成(靠自觉) ← 常用
强制性锁: 内核强制,没拿锁就读写会被阻塞/失败(很多系统如Ubuntu不支持)
```

**三个函数**:
```c
flock(fd, LOCK_EX|LOCK_NB);  // 只能锁【整个文件】, 只有建议性锁; LOCK_SH共享/LOCK_EX排它/LOCK_UN解锁/LOCK_NB非阻塞
fcntl(fd, F_SETLK, &flock);  // 可锁【文件的某段区域】, 支持建议+强制; 用struct flock(l_type/l_whence/l_start/l_len)
                             // F_RDLCK读锁(共享)/F_WRLCK写锁(独占)/F_UNLCK; F_SETLK非阻塞/F_SETLKW阻塞/F_GETLK测试
lockf(fd, ...);              // 是 fcntl 锁的封装
```
读锁共享、写锁独占(和读写锁一个道理)。了解即可,面试问得少。
