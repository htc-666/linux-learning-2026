# 第42章 新字符设备驱动实验 ⭐⭐⭐⭐⭐

> 正点原子《I.MX6U 嵌入式 Linux 驱动开发指南 V2.0.1》第四十二章  
> 本章不是重新学 LED 硬件，而是把第41章 LED 驱动改成 Linux 推荐的新字符设备写法：动态设备号、`cdev`、自动创建设备节点、文件私有数据。

---

## 本章总览

第42章 = **第41章 LED 寄存器点灯逻辑 + 新字符设备框架**。

第41章老写法：

```text
register_chrdev()
unregister_chrdev()
手动 mknod /dev/led c 200 0
```

第42章新写法：

```text
alloc_chrdev_region() / register_chrdev_region()
cdev_init()
cdev_add()
class_create()
device_create()
自动生成 /dev/newchrled
```

本章要解决第40/41章两个不优雅的地方：

| 老写法问题 | 第42章解决方法 |
|---|---|
| 主设备号手动写死，容易冲突 | 动态申请设备号 |
| `register_chrdev` 会占用整个主设备号下所有次设备号 | 用 `cdev_add` 精确添加指定数量设备 |
| 加载后还要手动 `mknod` | `class_create + device_create` 自动创建设备节点 |
| 设备属性散落成全局变量 | 封装成设备结构体，放到 `private_data` |

## 42.1 新字符设备驱动原理 ⭐⭐⭐⭐⭐

### 42.1.1 分配和释放设备号

第40章：

```c
register_chrdev(200, "chrdevbase", &fops);
```

问题：

```text
1. 要自己确定 200 没被占用
2. 一个主设备号下所有次设备号都被占用，浪费
```

第42章改为先申请设备号。

如果没有指定主设备号，用动态申请：

```c
int alloc_chrdev_region(dev_t *dev,
                        unsigned baseminor,
                        unsigned count,
                        const char *name);
```

如果已经指定主设备号，用静态注册：

```c
int register_chrdev_region(dev_t from,
                           unsigned count,
                           const char *name);
```

释放设备号统一用：

```c
void unregister_chrdev_region(dev_t from, unsigned count);
```

典型代码：

```c
if (newchrled.major) {
    newchrled.devid = MKDEV(newchrled.major, 0);
    register_chrdev_region(newchrled.devid, NEWCHRLED_CNT, NEWCHRLED_NAME);
} else {
    alloc_chrdev_region(&newchrled.devid, 0, NEWCHRLED_CNT, NEWCHRLED_NAME);
    newchrled.major = MAJOR(newchrled.devid);
    newchrled.minor = MINOR(newchrled.devid);
}
```

🔵 记忆：

```text
有主设备号 → register_chrdev_region
没主设备号 → alloc_chrdev_region
释放设备号 → unregister_chrdev_region
```

### 42.1.2 新的字符设备注册方法：`cdev`

Linux 用 `struct cdev` 表示一个字符设备。

```c
struct cdev {
    struct kobject kobj;
    struct module *owner;
    const struct file_operations *ops;
    struct list_head list;
    dev_t dev;
    unsigned int count;
};
```

核心成员：

| 成员 | 作用 |
|---|---|
| `ops` | 指向 `file_operations` |
| `dev` | 设备号 |
| `count` | 设备数量 |

初始化 `cdev`：

```c
cdev_init(&newchrled.cdev, &newchrled_fops);
newchrled.cdev.owner = THIS_MODULE;
```

添加字符设备：

```c
cdev_add(&newchrled.cdev, newchrled.devid, NEWCHRLED_CNT);
```

删除字符设备：

```c
cdev_del(&newchrled.cdev);
```

第42章完整注册链：

```text
申请设备号
    ↓
cdev_init 绑定 file_operations
    ↓
cdev_add 把字符设备添加进内核
```

第42章卸载链：

```text
cdev_del
    ↓
unregister_chrdev_region
```

## 42.2 自动创建设备节点 ⭐⭐⭐⭐⭐

这一节你一开始看不懂，是正常的。它讲的不是 LED 亮灭，而是：**驱动加载后，怎么自动生成 `/dev/newchrled`。**

第40/41章流程：

```bash
insmod led.ko
cat /proc/devices
mknod /dev/led c 200 0
./ledApp /dev/led 1
```

第42章希望变成：

```bash
insmod newchrled.ko
ls /dev/newchrled
./ledApp /dev/newchrled 1
```

也就是省掉手动 `mknod`。

### 42.2.1 mdev 机制

PC Linux 常用 `udev` 管理 `/dev` 设备节点。嵌入式 BusyBox 根文件系统常用简化版：`mdev`。

可以把 `mdev` 理解成：

```text
/dev 目录的自动管理员
```

驱动通过 `device_create()` 告诉内核“我有一个新设备”，内核产生热插拔事件，`mdev` 收到事件后自动创建 `/dev/newchrled`。

流程：

```text
驱动 device_create()
        ↓
内核产生 uevent 热插拔事件
        ↓
mdev 接收事件
        ↓
自动创建 /dev/newchrled
```

### 42.2.2 创建和删除类

先创建 class：

```c
newchrled.class = class_create(THIS_MODULE, NEWCHRLED_NAME);
```

可以理解为在 `/sys/class/` 下创建一个设备类别：

```text
/sys/class/newchrled/
```

删除 class：

```c
class_destroy(newchrled.class);
```

注意：`class_create()` 本身还不会创建 `/dev/newchrled`，它只是创建“类别”。

### 42.2.3 创建设备

真正触发 `/dev/newchrled` 自动生成的是：

```c
newchrled.device = device_create(newchrled.class,
                                 NULL,
                                 newchrled.devid,
                                 NULL,
                                 NEWCHRLED_NAME);
```

参数含义：

| 参数 | 含义 |
|---|---|
| `newchrled.class` | 设备属于哪个 class |
| `NULL` | 父设备，一般没有就填 NULL |
| `newchrled.devid` | 设备号 |
| `NULL` | 驱动私有数据，这里不用 |
| `NEWCHRLED_NAME` | 设备节点名字 |

效果：

```text
/dev/newchrled 自动出现
```

卸载时删除设备：

```c
device_destroy(newchrled.class, newchrled.devid);
```

完整创建顺序：

```text
cdev_add
    ↓
class_create
    ↓
device_create
    ↓
/dev/newchrled 自动生成
```

完整删除顺序：

```text
device_destroy
    ↓
class_destroy
    ↓
cdev_del
    ↓
unregister_chrdev_region
```

🔵 本节一句话：**`class_create + device_create` 的目的，就是替代手动 `mknod`，让驱动加载后自动生成 `/dev/xxx`。**

## 42.3 设置文件私有数据 ⭐⭐⭐⭐⭐

第40/41章里很多设备属性可能是散落的全局变量：

```c
dev_t devid;
struct cdev cdev;
struct class *class;
struct device *device;
int major;
int minor;
```

第42章把它们封装进一个设备结构体：

```c
struct newchrled_dev {
    dev_t devid;
    struct cdev cdev;
    struct class *class;
    struct device *device;
    int major;
    int minor;
};

struct newchrled_dev newchrled;
```

然后在 `open()` 里：

```c
filp->private_data = &newchrled;
```

这样后续 `read/write/release` 可以通过：

```c
struct newchrled_dev *dev = filp->private_data;
```

拿到设备结构体。

🔵 理解：`private_data` 是每次打开设备文件时，驱动给这个文件对象挂上的“自己的设备上下文”。真实驱动里非常常见。

## 42.5 实验程序编写 ⭐⭐⭐⭐⭐

本章例程：

```text
3_newchrled/newchrled.c
```

整体结构：

```text
头文件
宏定义：设备数量、设备名、LEDON/LEDOFF
寄存器物理地址宏定义
寄存器虚拟地址指针
newchrled_dev 设备结构体
led_switch()
led_open/read/write/release
file_operations
led_init()
led_exit()
module_init/module_exit
```

### LED 亮灭逻辑

`led_switch()` 和第41章一致：

```c
if (sta == LEDON) {
    val = readl(GPIO1_DR);
    val &= ~(1 << 3);
    writel(val, GPIO1_DR);
} else if (sta == LEDOFF) {
    val = readl(GPIO1_DR);
    val |= (1 << 3);
    writel(val, GPIO1_DR);
}
```

### `led_open()` 设置私有数据

```c
static int led_open(struct inode *inode, struct file *filp)
{
    filp->private_data = &newchrled;
    return 0;
}
```

### `led_write()` 接收用户命令

```c
retvalue = copy_from_user(databuf, buf, cnt);
ledstat = databuf[0];

if (ledstat == LEDON) {
    led_switch(LEDON);
} else if (ledstat == LEDOFF) {
    led_switch(LEDOFF);
}
```

### `led_init()` 的重点

分两部分：

1. 初始化 LED 寄存器：和第41章一样。
2. 使用新字符设备框架注册设备：第42章重点。

注册流程：

```c
alloc_chrdev_region(&newchrled.devid, 0, NEWCHRLED_CNT, NEWCHRLED_NAME);
newchrled.major = MAJOR(newchrled.devid);
newchrled.minor = MINOR(newchrled.devid);

newchrled.cdev.owner = THIS_MODULE;
cdev_init(&newchrled.cdev, &newchrled_fops);
cdev_add(&newchrled.cdev, newchrled.devid, NEWCHRLED_CNT);

newchrled.class = class_create(THIS_MODULE, NEWCHRLED_NAME);
newchrled.device = device_create(newchrled.class, NULL,
                                 newchrled.devid, NULL,
                                 NEWCHRLED_NAME);
```

### `led_exit()` 的重点

卸载时做相反操作：

```c
iounmap(...);

cdev_del(&newchrled.cdev);
unregister_chrdev_region(newchrled.devid, NEWCHRLED_CNT);

device_destroy(newchrled.class, newchrled.devid);
class_destroy(newchrled.class);
```

🔵 更严谨的顺序一般建议先销毁用户可见入口，再删 cdev 和设备号：

```text
device_destroy
class_destroy
cdev_del
unregister_chrdev_region
```

书上代码可运行，但后面自己写驱动时要养成错误处理和资源释放顺序意识。

## 本章个人理解与疑问记录

### 1. 42.2 到底在讲什么？

它不是讲怎么点灯，而是讲：

```text
驱动加载后如何自动生成 /dev/newchrled
```

第40/41章要手动：

```bash
mknod /dev/led c 200 0
```

第42章用：

```c
class_create();
device_create();
```

让 `mdev` 自动创建 `/dev/newchrled`。

### 2. `class_create` 和 `device_create` 分别干什么？

```text
class_create  → 创建设备类别，相当于 /sys/class/newchrled
device_create → 创建设备对象，触发 mdev 自动创建 /dev/newchrled
```

### 3. `cdev_add` 和 `device_create` 是不是重复？

不是。

```text
cdev_add       → 告诉内核：这个设备号对应哪些 file_operations
device_create  → 告诉用户空间：请在 /dev 下创建一个设备节点
```

一个负责内核分发，一个负责用户可见入口。

### 4. 第42章还需要关注 GPIO/时钟吗？

GPIO/时钟逻辑和第41章一样，不是第42章重点。第42章重点是新字符设备注册框架。

## 面试速查

**1. 第42章相比第40章新在哪里？**  
动态申请设备号、使用 `cdev` 注册字符设备、通过 `class_create/device_create` 自动创建设备节点。

**2. `alloc_chrdev_region` 和 `register_chrdev_region` 区别？**  
前者由内核动态分配设备号；后者注册开发者指定的设备号。

**3. `cdev_init` 做什么？**  
初始化 `struct cdev`，把 `cdev` 和 `file_operations` 绑定起来。

**4. `cdev_add` 做什么？**  
把初始化好的字符设备添加到内核，让内核能根据设备号找到对应 `file_operations`。

**5. `class_create/device_create` 的作用？**  
配合 `mdev/udev` 自动在 `/dev` 下创建设备节点。

**6. `private_data` 有什么用？**  
保存设备私有结构体指针，让 `read/write/release` 能拿到当前设备上下文。

**7. `cdev_add` 和 `/dev/newchrled` 是同一个东西吗？**  
不是。`cdev_add` 是内核里的字符设备注册；`/dev/newchrled` 是用户空间设备节点。
